# 로깅 전략

이 장에서는
서버 프로그램에서 **무엇을, 왜, 어떻게 기록(log)해야 하는지**의 기준을 정한다.

로깅은 단순한 출력이 아니라,
서버를 **운영 가능한 상태로 만드는 핵심 요소**다.

---

## 강의 목표

* 로그의 목적을 명확히 이해한다
* 디버깅용 출력과 로그의 차이를 구분할 수 있다
* 서버에서 반드시 남겨야 할 로그의 종류를 설명할 수 있다
* 과도하지 않고 일관된 로깅 기준을 이해한다

---

## 1. 로그란 무엇인가

로그(log)는
**서버에서 발생한 일을 시간 순서대로 기록한 정보**다.

서버에서 로그는 다음 역할을 한다.

* 장애 원인 추적
* 사용자 행동 분석
* 서버 상태 확인

> 로그는 "문제가 생겼을 때"만 보는 것이 아니다.

---

## 2. 디버깅 출력과 로그의 차이

초기 개발 단계에서는
`System.out.println()` 같은 출력으로도 충분해 보인다.

하지만 운영 서버에서는 문제가 된다.

| 디버깅 출력     | 로그        |
| ---------- | --------- |
| 개발 중 임시 확인 | 운영 기록 목적  |
| 형식 없음      | 형식 있음     |
| 환경별 제어 어려움 | 레벨별 제어 가능 |

---

## 3. 서버 로그의 기본 분류

서버 로그는 보통
중요도에 따라 나뉜다.

* **INFO**: 정상 흐름 기록
* **WARN**: 주의가 필요한 상황
* **ERROR**: 장애 또는 실패 상황

> DEBUG 레벨은 개발 환경에서만 사용한다.

---

## 4. 무엇을 로그로 남겨야 하는가

모든 것을 로그로 남기면
오히려 중요한 정보를 찾기 어렵다.

권장 로그 대상:

* 서버 시작/종료
* 주요 요청 처리 흐름
* 예외 발생 지점
* 비정상적인 상태 변화

---

## 5. 로그를 남기면 안 되는 것

다음 정보는 로그에 남기지 않는다.

* 비밀번호
* 인증 토큰
* 개인정보

> 로그는 외부로 유출될 수 있다는 전제로 작성해야 한다.

---

## 6. 일관된 로깅 기준의 중요성

로그 형식과 기준이 제각각이면
다음 문제가 생긴다.

* 검색이 어려움
* 자동 분석 불가
* 장애 대응 속도 저하

그래서 로그도
**규칙을 정하고 남긴다.**

---

## 7. 지금 단계에서 하지 않는 것

이 장에서는 다음을 다루지 않는다.

* 로그 프레임워크 설정
* 로그 포맷 상세 설정
* 파일 분리/로테이션

이 내용들은
운영 단계에서 다룬다.

---

## 이 장의 핵심 정리

* 로그는 운영을 위한 기록이다
* 출력과 로그는 다르다
* 로그는 레벨과 기준이 필요하다
* 민감 정보는 절대 로그에 남기지 않는다
